# PLAN-0319

Implementation plan for the next tooling and diagnostics iteration after closing `ex-pages/1a23-1`.

## Goals

1. Expand `ReadabilityCLI inspect` from candidate-scoring diagnostics into a broader extraction trace.
2. Improve observability before larger refactors.
3. Prepare a controlled path toward pipeline cleanup around leading associated content and site-rule phases.

## Scope Selection

Selected for implementation planning:

- Inspect enhancements:
  - sibling merge trace
  - site-rule trace
  - candidate context
  - content snapshot summary
  - selector probe
  - path / descriptor alignment work
- Library optimization work:
  - leading associated media rescue as an explicit pipeline phase
  - more explicit site-rule phase structure
  - review for source-DOM mutation across retry passes
  - review for site-specific fixes accidentally encoded as global heuristics
  - unify diagnostic language across tools

Not selected yet:

- broad reason-code rollout across every library path

## Batch Plan

### Batch 1 - Inspect Trace Foundation

#### Phase 1 - Data Model Expansion

Status: completed

Add internal/public report structures for:

- sibling decisions
- site-rule decisions
- candidate context
- content snapshot summary

Expected result:

- `parseWithInspection()` can carry more than candidate score data
- later formatter work does not require another report redesign

#### Phase 2 - Sibling Merge Trace

Status: completed

Record, per sibling under the selected candidate's parent:

- descriptor and DOM path
- visibility
- score, bonus, threshold
- final decision
- human-readable reason

Expected result:

- `inspect` can explain why leading wrappers, paragraphs, `BR`, or candidate siblings were appended or skipped

#### Phase 3 - Site-Rule Trace

Status: completed

Record deterministic site-rule decisions that explicitly return include/extract behavior in the sibling-merge path.

Expected result:

- `inspect` can show which rule fired, at which phase, on which node, and what it returned

### Batch 2 - Diagnostic Language Alignment

#### Phase 4 - Shared Path / Descriptor Formatting

Status: completed

Unify DOM path and descriptor generation used by inspection output and debugging helpers.

Expected result:

- candidate trace and DOM mismatch discussions use compatible terminology

### Batch 3 - Inspect Usability

#### Phase 5 - Content Snapshot Summary

Status: completed

Add compact summaries of the merged article content shape:

- selected candidate
- top-level child descriptors
- leading block descriptors
- content length

Polish completed:

- wrapper detection and wrapper path reporting
- article child count summary
- leading block summaries with path, child count, and text preview

#### Phase 6 - Selector Probe

Status: completed

Add `--selector` support to `inspect` for document-level targeted probing.

Expected result:

- fewer temporary scripts during debugging

### Batch 4 - Library Reviews And Refactors

Status: in progress

#### Track A - Structural Refactors

Status: completed

1. Introduce a dedicated leading-associated-content phase before generic sibling merge.
2. Make site-rule phase boundaries more explicit and more uniformly named.

#### Track B - Review Passes

Status: next

1. Review for source DOM mutations that leak across retry passes.
2. Review for global heuristics that should instead be site-specific rules.

#### Track C - Diagnostics Consistency

1. Reuse a common path / descriptor formatter across inspection and diff tooling.

## Recommended Execution Order

1. Batch 1 / Phase 1-3
2. Batch 2 / Phase 4
3. Batch 3 / Phase 6
4. Batch 3 / Phase 5 polish
5. Batch 4 / Track B reviews
6. Batch 4 / Track A refactors

Rationale:

- observability first
- unify terminology before adding more inspect surface area
- selector probing before snapshot polish because probe adds new debugging reach
- reviews second
- structural refactors last
